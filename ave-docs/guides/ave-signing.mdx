---
title: "Ave Signing"
description: "Create signature requests, collect user approvals, verify signatures, and handle expiry/denial paths."
---

Ave Signing provides identity-backed Ed25519 signatures for high-integrity user approval flows.

## Server flow

<Steps>
  <Step title="Create request">
    Call `POST /api/signing/request` with app credentials and operation payload.
  </Step>
  <Step title="Present signing UI">
    Show signing sheet/popup/embedded experience to the target identity.
  </Step>
  <Step title="Track request status">
    Poll `GET /api/signing/request/:requestId/status` until terminal state.
  </Step>
  <Step title="Verify signature before action">
    Verify via `POST /api/signing/verify` or local crypto verification before executing critical action.
  </Step>
</Steps>

## Create request

```json
{
  "clientId": "YOUR_CLIENT_ID",
  "clientSecret": "YOUR_CLIENT_SECRET",
  "identityId": "IDENTITY_UUID",
  "payload": "I approve this action",
  "metadata": { "action": "transfer" },
  "expiresInSeconds": 300
}
```

Constraints from implementation:

- Payload max 10,000 chars
- `expiresInSeconds` min 60, max 3600
- Identity must exist and have signing key

## Polling states

- `pending`
- `signed`
- `denied`
- `expired`

Server auto-marks expired requests when status endpoint is queried after expiry.

## Practical lifecycle sequence

1. Persist request record in your own DB with `pending` state
2. Surface UI timeout aligned with `expiresInSeconds`
3. Poll at short interval (for example 1-2s) with overall timeout
4. Transition local state to `signed` / `denied` / `expired`
5. Perform final verification before side effects

## Verify endpoint behavior

`POST /api/signing/verify` returns:

```json
{ "valid": true }
```

Invalid key/signature formats return `valid: false` with error string.

## Embed helpers

`@ave-id/embed` exposes:

- `openAveSigningSheet(...)`
- `openAveSigningPopup(...)`

Handle all terminal callbacks:

- Signed
- Denied
- Closed-without-action

## Edge cases

- Request belongs to wrong app (`403` on status)
- Request already resolved
- Request expired before user acts
- Identity has no signing key
- Signature provided but fails cryptographic verification
