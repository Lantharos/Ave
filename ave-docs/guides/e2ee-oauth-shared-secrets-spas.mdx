This page ties together **per-app encryption keys** (`#app_key`), **identity encryption keys** (ECDH for shared-secret / App key transfers), and **third-party SPA embeds** so you can ship one coherent integration.

## OAuth: E2EE and `app_key`

- Mark the app **E2EE-capable** in the developer portal (`supportsE2ee`).
- Request the scopes your product needs for identities and shared secrets (see [Scopes and claims](/guides/scopes-and-claims) and [App keys & shared secret transfers](/guides/shared-secret-transfers)).
- After authorization, the plaintext **app key** is delivered only in the redirect **URL fragment** as `app_key` — never in the JSON token body. Parse and normalize base64 (`+` vs space) per [End-to-end encryption](/guides/end-to-end-encryption).

## Identity encryption prerequisites

Before you can **create a transfer** to a handle, that identity must have a **public encryption key** registered so your client can run ECDH (`GET /api/encryption/public-key/:handle`).

### `wrap_mode: master` (Ave web default)

- `encryptedPrivateKey` is an opaque base64 blob produced by Ave clients (AES-GCM wrapping PKCS#8 with the user’s **master key**).
- `POST /api/encryption/keys/:identityId` with default `wrap_mode` (or `master`) returns **409** if a key row already exists.

### `wrap_mode: oauth_app_key` (third-party wrap)

SPAs often wrap the identity PKCS#8 with a key derived from the OAuth **app key** instead of the Ave master key. Ave **does not unwrap** this blob: it only checks **base64 validity** and **decoded size bounds** (same checks apply to `publicKey`).

Request shape:

```json
{
  "wrap_mode": "oauth_app_key",
  "oauth_app_id": "OAUTH_APP_UUID",
  "device_key": "optional-device-slot",
  "publicKey": "BASE64_SPKI",
  "encryptedPrivateKey": "YOUR_AES_GCM_ENVELOPE"
}
```

- **`oauth_app_id`** must match a real app row in Ave (validated server-side).
- **`device_key`**: optional, default `default`. Use another value (e.g. `browser-2`) to store **additional** ciphertexts for the same keypair and app when the primary row is already used.

### Multi-device when Ave already has a master-wrapped key

If the user registered keys on **aveid.net** first (`wrap_mode: master`), a naive `POST` with `oauth_app_key` used to collide with **409**. Now:

- If the identity already has a **master** primary row, a compatible `POST` with `oauth_app_key` **upserts** an **OAuth wrap** row as long as **`publicKey` matches** the existing SPKI.
- Additional devices for the same OAuth app: send the **same `publicKey`** and a **distinct `device_key`** per device (or reuse `default` only for the primary OAuth ciphertext on the main row when the primary was created as oauth-only).

## Shared secret recipient storage formats

`POST /api/shared-secrets/:id/finalize-recipient-storage` always stores an **opaque** `encryptedSecretForRecipient`. Your app must use the **same wrap** when reading via `GET /api/shared-secrets/access` (e.g. HKDF + AES-GCM in the client).

### Hosted claim page (`/shared/claim`)

Default behavior: after ECDH decrypt, Ave re-wraps with the user’s **master key** (same as first-party Ave web).

For apps that wrap recipient storage with **HKDF from the OAuth app key** (salt `citadel:ss:${resourceKey}` — see implementation below), add query flags:

| Query param | Purpose |
|-------------|---------|
| `finalize_wrap=citadel_v1` | Re-wrap decrypted plaintext with the Citadel-compatible storage key instead of the master key. |
| `resource_key=...` | Optional override for the salt segment when the secret has no `resourceKey`. |
| `defer_finalize=1` | **Embed only.** Ave runs `claim` + decrypt but **does not** call finalize; your host calls `claimAndStoreSharedSecret` (or equivalent) once with matching `wrapForRecipientStorage`. |

#### `citadel_v1` KDF (reference)

Aligned with Ave web `deriveCitadelRecipientStorageKeyFromAppKey`:

- **Salt** (UTF-8): `citadel:ss:${resourceKey}`
- **IKM**: raw bytes of the OAuth app key (after base64 decode, with `+` normalization).
- **HKDF**: SHA-256, Web Crypto `deriveKey` with **info** (UTF-8): `citadel-recipient-wrap-v1`
- **Output**: AES-GCM 256-bit key; encrypt the plaintext secret with a random 12-byte IV and store **IV ‖ ciphertext** as base64 (same `encrypt` pattern as other Ave client crypto).

The host must supply the app key to the iframe via **`postMessage`**:

```json
{ "type": "ave:provide_app_key", "payload": { "appKeyB64": "..." } }
```

`@ave-id/embed` can send this automatically when you pass `finalizeRecipientWrap: "citadel_v1"` and `appKeyB64` into `openAppKeyClaimSheet` / `openAppKeyClaimPopup`.

### Recommended embed sequences

1. **`openAppKeyClaimSheet` + default finalize** — Simplest if your app reads recipient blobs with the **Ave master key** in a first-party context (rare for pure SPAs).

2. **`openAppKeyClaimSheet` + `finalize_wrap=citadel_v1` + `appKeyB64`** — Hosted UX while keeping **recipient storage** compatible with app-key-derived wraps (`getSharedSecretAccess` + `openBytesSealedWithAppKey`-style flows).

3. **`defer_finalize=1` + in-app `claimAndStoreSharedSecret`** — Single finalize, no double-wrap risk: the iframe only authenticates and proves the user can decrypt; your app performs **one** claim + **one** finalize with your `wrapForRecipientStorage` callback. Listen for `payload.kind === "app_key_claim_deferred"` on success.

## API quick reference

- `GET /api/encryption/keys/:identityId` — Returns `wrapMode`, optional `oauthAppId`, and `oauthWraps[]` (per-app, per-`device_key` ciphertexts).
- `POST /api/encryption/keys/:identityId` — Master or OAuth wrap semantics above; responses may include `wrapMode: "oauth_wrap_added"` when adding a side wrap.
- [App keys (shared secret) API](/api/shared-secret-endpoints) — Transfers, claim, finalize.

## Operational notes

- Treat **claim URLs** as bearer secrets.
- Identity private key material is still decrypted inside the hosted claim page using the **master key** today; users who only have OAuth-wrapped identity keys should prefer **in-app** claim until Ave supports supplying PKCS#8 through a vetted embed path.
- Apply database migration **`0011_identity_encryption_wrap_modes`** before relying on new columns/tables in production.
