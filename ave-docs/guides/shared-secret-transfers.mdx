---
title: "App keys & shared secret transfers"
description: "How apps grant encrypted vault or workspace keys to another Ave identity—without plaintext on the server."
---

These transfers power **App keys** in the Ave dashboard (the dashboard only **lists** what exists; **your app** creates transfers using the API or SDK after sign-in).

Ave shared secret transfers let one identity package a secret for another Ave handle and hand them a claim link. Ave stores only encrypted blobs:

- the founder stores their own wrapped copy in Ave
- the founder encrypts a transfer copy to the recipient identity's encryption public key
- the recipient claims the transfer, decrypts locally, and re-wraps it for their own Ave storage

This is useful for:

- vault roots or workspace keys in apps like Citadel
- multiple per-app keys identified by `resourceKey`
- app-defined team or room keys that should persist in Ave per identity

## What Ave adds

Ave now has a dedicated **identity encryption keypair** per identity, separate from Ave Signing.

- `publicKey` is published for lookup by handle
- `encryptedPrivateKey` is stored encrypted with the user's master key
- the server never receives the plaintext private key or plaintext shared secret

## Transfer flow

<Steps>
  <Step title="1. Your app creates the shared secret">
    The app (with the user signed in) creates a shared secret record and stores the owner’s wrapped copy in Ave—usually via `createSharedSecretWithTransfer` or `POST /api/shared-secrets` plus a transfer.
  </Step>
  <Step title="2. Founder looks up recipient encryption key">
    The app calls `GET /api/encryption/public-key/:handle` to get the recipient identity's public encryption key.
  </Step>
  <Step title="3. Founder encrypts for recipient locally">
    The app encrypts the secret to the recipient's public key client-side and sends only the ciphertext to Ave when creating the transfer contract.
  </Step>
  <Step title="4. Founder sends claim link">
    Ave returns a claim URL. The founder can send it in chat, email, or any app-specific invite flow.
  </Step>
  <Step title="5. Recipient claims as matching handle">
    The recipient opens the link, signs into Ave, and the authenticated identity handle must match the contract target handle.
  </Step>
  <Step title="6. Recipient stores their own copy">
    The recipient decrypts the transfer locally, re-encrypts it with their Ave master key, and finalizes recipient storage in Ave.
  </Step>
</Steps>

## Secret types

### Global secrets

Use `kind: "global"` for secrets that are not tied to a specific OAuth app record.

Examples:

- a manually managed vault root
- an out-of-band team bootstrap secret
- a cross-app workspace secret

### App-scoped secrets

Use `kind: "app_scoped"` plus `appId` when the secret belongs to a specific Ave app.

Use `resourceKey` when the app has multiple stable secrets.

Examples:

- `vault.root`
- `workspace.main`
- `channel:general`

## Security model

- founder and recipient identities are isolated
- the recipient must already exist as an Ave handle in v1
- claim links are bearer-style secrets, so treat them as sensitive
- claim tokens expire and cannot be reused after claim
- recipient storage is identity-specific and should never be merged across identities

<Warning>
Do not use a handle as your long-term database key in your own app. Use the stable Ave identity ID. Handles are for lookup and user-facing targeting.
</Warning>

## Recommended app pattern

1. Store your own app's encrypted data using the secret retrieved from Ave.
2. Keep one secret descriptor per logical key domain.
3. Use `resourceKey` to distinguish multiple per-app keys.
4. Re-fetch recipient access from Ave after the recipient claims the transfer.

Do **not** rely on users to “share keys” from the Ave website—implement invite/grant flows inside your product using `createSharedSecretWithTransfer` (or the lower-level endpoints).

## Founder offline, teammates inviting others

- **First invite (founder → teammate)** — Once the founder creates a transfer, only Ave and ciphertext are involved; the founder does not need to stay online for the recipient to open the claim link and finish.
- **Further invites (teammate → teammate)** — Anyone who has **already claimed** recipient access for that shared secret can create **additional** transfers to new handles (`POST /api/shared-secrets/:id/transfers` with their own `identityId`). They encrypt the plaintext locally (same as the founder) using `createSharedSecretPackage` + `createSharedSecretTransfer` from `@ave-id/sdk` / `@ave-id/sdk/client`. The founder does not need to be online.

## Embedded claim UI (Citadel-style)

Host apps can open the Ave claim experience in a **sheet or popup** so users never leave your product:

```ts
import { openAppKeyClaimSheet } from "@ave-id/embed";

openAppKeyClaimSheet({
  claimUrl, // from your transfer API response, or use claimToken
  onSuccess: (payload) => {
    // payload.kind === "app_key_claim"
    // payload.redirectUrl — optional callback URL with shared_secret_* query params
    // payload.sharedSecretId, transferId, identityId, handle
  },
});
```

The iframe loads `/shared/claim?token=...&embed=1` on `aveid.net` and posts `ave:success` to your page when the user finishes. Same pattern as signing embeds.

See [@ave-id/embed](/sdk/embed-sdk#app-key-claim-embeds).

## SDK implementation

This is the full browser-side flow using the SDK rather than hand-calling endpoints.

Use this when your app already has:

- an Ave session token
- the current Ave identity ID
- a secret you want to share
- a way to encrypt the founder's own stored copy
- a way to encrypt the recipient's stored copy after claim

### Founder flow

Create the shared secret and transfer contract in one SDK call:

```ts
import { createSharedSecretWithTransfer } from "@ave-id/sdk/client";

const result = await createSharedSecretWithTransfer({
  issuer: "https://aveid.net",
  sessionToken,
  ownerIdentityId,
  targetHandle: "alice",
  kind: "app_scoped",
  appId,
  resourceKey: "vault.root",
  label: "Citadel Vault Root Key",
  returnUrl: "https://citadel.app/invite/accepted",

  // Your app decides how to wrap the founder's own copy.
  encryptedSecretForOwner,

  // Plaintext secret stays local; SDK encrypts it for the recipient.
  secret: vaultRootPlaintext,
});

console.log(result.claimUrl);
```

What the SDK does:

1. creates the shared secret record in Ave
2. looks up the recipient identity encryption public key
3. encrypts the plaintext secret locally for the recipient
4. creates the transfer contract
5. returns the claim URL

If `returnUrl` is set, Ave will redirect the recipient back there after a successful claim and append:

- `shared_secret_claimed=1`
- `shared_secret_id=...`
- `transfer_id=...`
- `identity_id=...`

That lets your app treat the claim as both:

- secret access being granted
- invite acceptance or organization join confirmation

### Recipient flow

Claim and finalize through the SDK:

```ts
import {
  claimAndStoreSharedSecret,
  importIdentityEncryptionPrivateKey,
} from "@ave-id/sdk/client";

const recipientPrivateKey = await importIdentityEncryptionPrivateKey(
  recipientPrivateKeyPkcs8Base64
);

await claimAndStoreSharedSecret({
  issuer: "https://aveid.net",
  sessionToken,
  claimToken,
  recipientIdentityId,
  recipientPrivateKey,

  async wrapForRecipientStorage(secretBytes, transfer) {
    return await encryptForYourStorage(secretBytes, transfer.descriptor);
  },
});
```

What the SDK does:

1. validates the claim token against the authenticated identity
2. fetches the encrypted transfer payload
3. decrypts it locally with the recipient identity private key
4. calls your storage-wrapping function
5. finalizes the recipient-owned envelope in Ave

After the redirect, your app can call `getSharedSecretAccess(...)` to load the recipient envelope for future use. On later logins or future auth events, the app can fetch the same envelope again without requiring another claim.

### Lower-level SDK helpers

If you want more control, use:

- `getIdentityEncryptionPublicKey`
- `createSharedSecret`
- `createSharedSecretTransfer`
- `resolveSharedSecretTransfer`
- `claimSharedSecretTransfer`
- `finalizeSharedSecretRecipientStorage`
- `generateIdentityEncryptionKeyPair`
- `encryptSharedSecretForIdentity`
- `decryptSharedSecretFromTransfer`

### Storage strategy note

Ave stores encrypted blobs exactly as your app provides them. Your app still chooses the wrapping strategy for:

- the founder's stored copy
- the recipient's stored copy after claim

Common patterns:

- encrypt with your app's existing Ave-delivered app key
- encrypt with an app-managed workspace key
- encrypt with a local key derived from a broader vault context

### Recommended `resourceKey` scheme

Use stable keys such as:

- `vault.root`
- `workspace.main`
- `room.general`
- `channel:abc123`

## Current v1 limits

- existing Ave handles only
- share and claim only
- no built-in revoke or rotate workflow yet
- no first-class multi-member group object yet (use chained transfers + your app’s membership model)

If you need full membership lifecycle, build it in your app on top of these primitives first, then promote stable patterns into Ave later.
