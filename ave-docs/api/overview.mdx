## Base URL

```
https://api.aveid.net
```

<Note>
  Use `api.aveid.net` for all API calls. Use `aveid.net` for user-facing auth pages (sign-in, Connector consent). Never mix the two.
</Note>

## Request conventions

All request bodies are JSON. The token endpoint accepts both OAuth-standard `snake_case` fields and legacy `camelCase` fields.

<CodeGroup>
```bash Token request
curl -X POST https://api.aveid.net/api/oauth/token \
  -H "Content-Type: application/json" \
  -d '{
    "grant_type": "authorization_code",
    "code": "AUTHORIZATION_CODE",
    "redirect_uri": "https://yourapp.com/callback",
    "client_id": "YOUR_CLIENT_ID",
    "code_verifier": "PKCE_VERIFIER"
  }'
```

```bash Userinfo request
curl https://api.aveid.net/api/oauth/userinfo \
  -H "Authorization: Bearer ACCESS_TOKEN"
```
</CodeGroup>

## Authentication by endpoint family

<AccordionGroup>
  <Accordion title="Public OAuth/OIDC endpoints ‚Äî no auth required">
    These endpoints are public and require no credentials:

    - `GET /.well-known/openid-configuration` ‚Äî OIDC discovery metadata
    - `GET /.well-known/jwks.json` ‚Äî JWT verification keys
    - `GET /.well-known/webfinger` ‚Äî issuer discovery
    - `GET /api/oauth/app/:clientId` ‚Äî app public metadata
    - `GET /api/oauth/resource/:resourceKey` ‚Äî Connector resource metadata
    - `POST /api/oauth/token` ‚Äî token grants (credentials validated in request body)

    Userinfo requires a bearer token:
    - `GET /api/oauth/userinfo` ‚Äî requires `Authorization: Bearer <access_token>`
  </Accordion>

  <Accordion title="Session-authenticated OAuth management endpoints">
    These require an authenticated Ave session (bearer session token or Ave session cookie):

    - `POST /api/oauth/authorize` ‚Äî grant authorization and produce a redirect URL
    - `GET /api/oauth/authorizations` ‚Äî list active authorizations
    - `GET /api/oauth/authorization/:clientId` ‚Äî get authorization for a specific app
    - `GET /api/oauth/delegations` ‚Äî list active Connector grants
    - `DELETE /api/oauth/delegations/:delegationId` ‚Äî revoke a Connector grant
  </Accordion>

  <Accordion title="Signing endpoints ‚Äî app credentials in body">
    App-facing signing endpoints authenticate with `clientId` + `clientSecret` in the request body:

    - `POST /api/signing/request` ‚Äî create a signing request
    - `GET /api/signing/request/:requestId/status` ‚Äî poll request status
    - `GET /api/signing/public-key/:handle` ‚Äî look up an identity's public key
    - `POST /api/signing/verify` ‚Äî verify a signature

    User-facing signing endpoints require a session token:
    - `GET /api/signing/keys` ‚Äî list the user's signing keys
    - `POST /api/signing/keys/:identityId` ‚Äî create a signing key
    - `PUT /api/signing/keys/:identityId` ‚Äî rotate a signing key
    - `POST /api/signing/requests/:requestId/sign` ‚Äî approve a request
    - `POST /api/signing/requests/:requestId/deny` ‚Äî deny a request
  </Accordion>

  <Accordion title="Identity key endpoints ‚Äî unified with signing">
    Identity keys are now the same keypair concept used by signing.

    Public lookup endpoints:

    - `GET /api/signing/public-key/:handle` ‚Äî canonical identity public key lookup
    - `GET /api/encryption/public-key/:handle` ‚Äî compatibility alias

    Session-authenticated key envelope endpoints:

    - `GET /api/signing/keys/:identityId`
    - `POST /api/signing/keys/:identityId`
    - `PUT /api/signing/keys/:identityId`
    - `GET /api/encryption/keys/:identityId` (alias)
    - `POST /api/encryption/keys/:identityId` (alias)
    - `PUT /api/encryption/keys/:identityId` (alias)
  </Accordion>
</AccordionGroup>

## CORS behavior

OAuth and OIDC paths are CORS-enabled with controlled origin resolution. Session-protected mutating endpoints with cookies enforce origin allowlisting.

<Warning>
  If you use cookie-based session auth for mutating endpoints, requests from disallowed origins are rejected.
</Warning>
